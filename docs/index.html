Design Rationale

Why we built it this way

Why 4 Levels?

Standard SDLC works for monolithic apps, but SuperApps contain 5-10+ mini-apps with 10-50+ features each. Different teams own different SubApps and need isolated documentation.

L0: Platform Entire app, shared infrastructure
L1: SubApp Single mini-app domain
L2: Epic User-facing capability
L3: Feature Single implementable behavior

Why Cascading FR?

Requirements often get "lost" between high-level goals and implementation. Cascading traceability ensures every platform requirement reaches code.

Platform FR cpt-platform-fr-offline
↓
SubApp FR cpt-learn-fr-offline-courses
↓
Feature Flow cpt-learn-flow-load-cached
↓
Code Marker @cpt-flow:...:p1

Why MVI Pattern?

After evaluating MVVM, MVC, Redux — MVI was chosen for unidirectional data flow, cross-platform consistency, and direct mapping from CDSL flows to Intents.

State → UI → Intent → Reducer → State

Why Block Markers?

Simple @cpt-impl shows "this file implements X". Block markers (@cpt-begin/@cpt-end) show which exact lines implement which CDSL instruction.

  • Checkbox Cascade: all markers exist → mark step [x]
  • Code Review: see which design step each block implements
  • Refactoring Safety: markers stay with the logic

Evolution from SDLC Kit

This kit started as an extension of the standard SDLC Kit, then diverged to address mobile-specific needs: 4-level hierarchy for SuperApp scale, MVI pattern for state management, platform-specific implementation sections for KMP/Android/iOS separation.
